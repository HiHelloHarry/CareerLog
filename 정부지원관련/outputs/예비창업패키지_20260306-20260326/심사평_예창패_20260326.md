## 심사위원 A 평가 (69점) -- 투자심사역 출신

### 문제인식 (20/25)
설문(n=34)으로 문제 존재를 확인한 점은 좋다. "88%가 상시 관리 안 함", "62%가 하루 이상 소요" 등 수치가 구체적이고 문제-솔루션 연결이 명확하다. 다만 n=34는 통계적 유의성이 약하다. 이 표본이 대표성을 갖는다고 주장하기엔 부족하고, 설문 대상이 지인 중심인지 비지인인지 명시되지 않았다. "국내외에 없다"는 카테고리 독점 주장은 MS Recall, Rewind AI와의 차이를 더 날카롭게 정리해야 설득력이 있다.

### 실현가능성 (18/25)
MVP v0.8 완성은 예비창업자로서 매우 우수하다. GitHub 공개 저장소 링크가 있어 검증 가능하다는 것도 플러스. 그러나 협약기간 내 목표가 "베타 100명 + 결제 시스템 + Mac 버전"으로 과다하다. 1인 개발자가 베타 운영, 외주 관리, Mac 포팅을 동시에 하겠다는 건 비현실적이다. 외주 개발 관리 경험이 있는지도 불분명하다.

### 성장전략 (16/25)
**치명적 수치 오류 발견:**
1. 18개월 MRR 500만원인데 바로 옆에 "ARR 약 6억원"으로 표기. MRR 500만 x 12 = 6,000만원이지, 6억이 아니다. 10배 차이.
2. SOM에서도 "유료 약 1,000명, ARR 약 6억원"이라 했는데, 1,000명 x 연 58,800원 = 5,880만원. 이것도 6억이 아닌 약 5,900만원이다.
3. 18개월 MAU 10,000에 유료전환율 7% 적용하면 유료 700명, MRR = 700 x 4,900 = 343만원인데 목표가 500만원이다. 유료전환율이 10% 이상이어야 달성 가능한데, 7%로 명시해놓고 500만원을 쓰면 모순이다.

수익 모델 자체(Free-Pro-Premium 3티어)는 표준적이고 이해하기 쉽다. 단위 경제(마진율 69%, BEP 300명)도 명시한 점은 좋으나, 위 수치 오류가 신뢰를 심각하게 훼손한다.

### 팀구성 (15/25)
1인 창업자의 실행력은 인정한다. MVP를 혼자 만든 건 대단하다. 그러나 "기술 고문 예정('26.5)", "공동창업자 탐색 중(후보 2명 대화 진행 중)"은 확정이 아니라 희망사항이다. 투자심사 관점에서 1인팀은 리스크가 높고, "예정"과 "탐색 중"은 가점이 되지 않는다.

**총평**: 문제 인식과 MVP 완성도는 예비창업패키지 지원자 중 상위권이다. 그러나 핵심 수치(ARR, MRR, SOM)에 산술 오류가 3건 이상 발견되어 사업 계획의 신뢰성에 의문이 생긴다. 심사 테이블에서 이런 오류가 발견되면 "숫자를 대충 적었다"는 인상을 주고, 다른 수치도 의심하게 된다. 반드시 제출 전 수정해야 한다.

**합격 의견**: 반대 (수치 오류 수정 전제 시 찬성으로 전환 가능)

## 심사위원 B 평가 (73점) -- 기술 전문가

### 문제인식 (21/25)
문제 정의가 구조적("귀찮아서 안 한다")으로 잘 프레이밍되어 있다. 기존 솔루션 비교 테이블이 깔끔하고, 각 솔루션의 근본 한계를 정확히 짚었다. 설문 데이터도 문제 정의와 1:1 대응시켜 서술한 점이 좋다. 다만 "국내외에 없다"는 주장에 대해 ActivityWatch(오픈소스), WakaTime(개발자용) 같은 자동 추적 도구가 존재한다. 이들과의 차별점을 명시하지 않은 것은 아쉽다.

### 실현가능성 (19/25)
기술적으로 인상적인 부분이 많다:
- Win32 API 직접 호출로 네이티브 모듈 의존 제거 -- Electron 패키징 문제를 우회한 실용적 판단
- 앱별 파서 17개, 브라우저 URL 추출(UIAutomation), 키보드/마우스 활동 감지, Flow 감지 엔진
- UIAutomation 창 제목 오염 버그 발견/수정 경험 -- 실제 구현 역량 증명

궁금한 점:
1. "파싱 성공률 100%"는 가상 시뮬레이션 결과라 했는데, 실제 유저 환경에서의 에지 케이스(멀티모니터, 가상 데스크톱, 관리자 권한 앱 등)는 어떻게 처리하는가?
2. Mac 버전에서 osascript 기반 활동 감지가 Windows의 Win32 API와 동등한 수준의 정확도를 달성할 수 있는가? macOS의 보안 정책(Accessibility 권한, Screen Recording 권한 등)으로 인한 제약은?
3. 10초 폴링 방식의 배터리/CPU 영향은? 장시간 사용 시 성능 데이터가 있는가?
4. 로컬 저장이라 했지만, AI 변환 시에는 서버로 데이터가 전송되는 구조다. 이 부분의 프라이버시 설계는?

기술 로드맵에서 지재권 전략이 보이지 않는다. "앱별 파서 17개 + Flow 감지 엔진"은 특허 출원 가능한 영역이다. 기술적 해자를 주장하면서 지재권 확보 계획이 없는 것은 모순이다.

### 성장전략 (17/25)
기술적 관점에서 단위 경제 계산은 합리적이다(AI 비용 1회 $0.03, 마진율 69%). 다만 심사위원 A가 지적한 MRR/ARR 산술 오류는 동의한다. 기술자 입장에서도 숫자가 맞지 않으면 나머지 기술 수치의 신뢰도에도 영향을 준다.

Mac 버전 개발을 "외주 개발자 단기 계약 2개월"로 완료하겠다는 계획은 비현실적이다. Windows에서 Win32 API로 구현한 활동 감지 엔진을 macOS로 포팅하는 건 단순 작업이 아니다. osascript의 한계, Accessibility 권한 문제, AppleScript와 JXA 선택 등 기술적 의사결정이 필요하고, 이를 외주에 맡기면 품질 관리가 어렵다.

### 팀구성 (16/25)
대표자의 기술 역량은 MVP로 충분히 증명됐다. 기획/QA/PM 배경에서 출발해 Electron + Win32 API + AI 프롬프트 엔지니어링까지 독학한 것으로 보이며, 이는 높이 평가한다. 그러나 "기술 고문 비상근"으로는 코드 리뷰 정도만 가능하고, 아키텍처 의사결정의 속도가 느릴 수밖에 없다. Mac 포팅처럼 기술적 난이도가 높은 과제에서 CTO급 인력 부재가 병목이 될 것이다.

**총평**: 기술 완성도는 예비창업 단계에서 보기 드문 수준이다. 특히 네이티브 모듈 의존 없이 Win32 API로 직접 구현한 판단력, 앱별 파서 17개를 직접 작성한 실행력이 인상적이다. 지재권 전략 부재와 Mac 포팅 계획의 비현실성이 주요 감점 요인이다.

**합격 의견**: 찬성 (조건: 수치 오류 수정, 지재권 전략 추가)

## 심사위원 C 평가 (74점) -- 창업 멘토

### 문제인식 (22/25)
창업자 본인이 이 문제를 직접 경험했고, 그 경험에서 출발해 제품을 만들었다는 것이 느껴진다. "기획/QA/PM 멀티롤로 실무 수행하며 이직 준비 시 경력 정리가 안 돼 있었다"는 진정성이 있다. 설문 n=34는 작지만, 최소한 설문을 기획하고 실행했다는 것 자체가 고객 검증 의지를 보여준다. 특히 "바로 써보고 싶다" 응답자 10명의 연락처를 확보했다는 점은 좋다. 이 10명이 실제로 앱을 설치하고 2주 이상 사용한 데이터가 있었다면 만점에 가까웠을 것이다.

한 가지 아쉬운 점: 설문이 2026.03.25(어제)에 실시됐다. 제출 직전에 급하게 한 것 아닌가 하는 인상을 줄 수 있다. 심사위원에 따라 "설문을 넣기 위한 설문"으로 보일 수 있으니 유의해야 한다.

### 실현가능성 (18/25)
MVP가 완성되어 있고, Setup.exe로 배포 가능하다는 것은 예비창업패키지에서 매우 강력한 무기다. 대부분의 지원자가 "만들겠습니다"인데 이미 만들어놓은 것은 차별화된다. GitHub 링크도 좋다.

다만 실제 사용자 데이터가 없다. "가상 유저 시뮬레이션"은 말 그대로 시뮬레이션이지 실사용 검증이 아니다. 연락처 확보한 10명에게 지금이라도 앱을 보내고, 3일간의 사용 후기라도 확보하면 심사에서 훨씬 유리하다.

협약기간 내 계획에서 베타 100명과 Mac 버전 동시 추진은 부담스럽다. 연쇄창업 경험에서 배운 것은, 초기에는 하나에 집중해야 한다는 것이다. Windows 베타에서 PMF를 확인한 뒤 Mac으로 가는 게 맞지, 동시에 가면 양쪽 다 중도반단 위험이 있다.

### 성장전략 (18/25)
"매일 업무 시작/종료와 함께 쓰는 습관 도구 -> 기록 축적 -> 이탈 비용 상승 -> 유료 전환"이라는 플라이휠이 논리적이다. 설문에서 유료 의향 71%, 5,000원 이하 수용 71%를 근거로 4,900원을 설정한 것도 나쁘지 않다.

그러나 몇 가지 현실적 우려가 있다:
1. **수치 오류**: 다른 심사위원들도 지적했지만, ARR 6억과 MRR 500만의 불일치는 심각하다. 창업자가 자기 사업의 숫자를 정확히 모르면 신뢰가 무너진다.
2. **유료 전환 경로의 모호함**: Free에서 Pro로 넘어가는 트리거가 "세션 20개 소진"인데, 매일 1세션이면 20일 만에 벽에 부딪힌다. 이때 사용자가 돈을 내게 할 설득 장치(예: 무료 체험 중 Pro 기능 일부 노출)가 구체적이지 않다.
3. **채용 플랫폼 파트너십**: "잡코리아/원티드 연동"은 27년 하반기 계획인데, 어떤 형태의 파트너십인지, 사전 접촉이 있는지 불명확하다. 희망사항에 가깝다.

GTM에서 "블라인드 이직 카테고리에 실사용 후기 작성"은 현실적이고 좋은 전략이다. 돈 안 들고, 타겟이 정확하고, 창업자가 직접 할 수 있다.

### 팀구성 (16/25)
창업자의 진정성과 실행력은 의심의 여지가 없다. 혼자서 아이디어부터 배포까지 전부 해냈다. 이런 사람은 드물다.

그러나 사업을 키우는 것과 제품을 만드는 것은 다른 역량이다. "그로스 공동창업자 탐색 중(후보 2명 대화 진행 중)"이라 했는데, 이 후보들의 배경이나 확보 가능성에 대한 정보가 없다. "지분 5~15% 협의"라고 적었는데, 이 범위가 넓어서 아직 구체적 논의가 시작되지 않은 것으로 보인다.

제안: 공동창업자가 아직 없다면 솔직하게 쓰되, "그래서 내가 직접 GTM을 이렇게 하겠다"는 구체적 실행안으로 보완하는 게 낫다. 없는 팀원을 있는 척하는 것보다, 혼자서도 할 수 있는 범위를 현실적으로 보여주는 게 설득력 있다.

**총평**: 창업자의 진정성, 실행력, 고객 검증 의지 모두 좋다. MVP가 완성되어 있다는 것은 예비창업패키지에서 매우 강한 카드다. 다만 수치 오류가 여러 곳에서 발견되어, 심사 테이블에서 신뢰를 잃을 위험이 있다. 제출 전 반드시 수정하라. 그리고 가능하다면 10명의 시드 유저에게 실제로 앱을 써보게 하고, 짧은 후기라도 확보해서 넣어라. "만들었습니다"와 "쓰고 있습니다"는 심사위원 인식에서 하늘과 땅 차이다.

**합격 의견**: 찬성

## 핵심 피드백 요약

### 강점
1. **MVP v0.8 완성 + 즉시 배포 가능**: 예비창업 단계에서 GitHub 공개 소스코드와 Setup.exe 인스톨러까지 완성한 것은 상위 5% 수준의 실행력이다.
2. **문제-솔루션 프레이밍이 날카롭다**: "귀찮아서 안 한다 -> 귀찮음을 구조적으로 제거한다"는 명확한 메시지. 설문 데이터와 1:1 대응시킨 서술도 좋다.
3. **기술적 깊이와 독자적 구현**: Win32 API 직접 호출, 앱별 파서 17개, Flow 감지 엔진 등 단순 래핑이 아닌 자체 구현. UIAutomation 버그 발견/수정 경험은 실력의 증거다.

### 보완 필수 항목 (제출 전 수정 필수)

1. **[수치 일관성] ARR/MRR/SOM 산술 오류 -- 가장 치명적**
   - 문제점: (1) MRR 500만 x 12 = 6,000만원인데 "ARR 약 6억"으로 10배 오기재. (2) SOM "유료 1,000명, ARR 약 6억"도 1,000 x 58,800 = 5,880만원으로 불일치. (3) MAU 10,000 x 유료전환율 7% = 700명인데 MRR 500만원 달성하려면 1,020명(전환율 10.2%) 필요.
   - 권장 수정: ARR "약 6억"을 모두 "약 6,000만원"으로 수정하거나, SOM 목표를 재조정. 유료전환율 또는 MRR 목표 중 하나를 수정하여 산술적 일관성 확보. 모든 수치를 하나의 스프레드시트에서 산출하고, 역산 검증(MRR = 유료 수 x 단가, ARR = MRR x 12) 실시.

2. **[실사용 검증 부재] 가상 시뮬레이션만으로는 부족**
   - 문제점: "가상 유저 3개 페르소나 시뮬레이션"은 실제 사용자 검증이 아니다. 시드 유저 10명의 연락처를 확보했다고 했으나, 아직 1명도 실제로 앱을 사용하지 않았다.
   - 권장 수정: 제출 마감 전까지 시드 유저 3~5명에게 앱을 배포하고, 최소 3일간 사용 후기(스크린샷 + 한 줄 코멘트)를 확보하여 첨부. "써보고 싶다"에서 "써보고 있다"로 전환.

3. **[지재권 전략 부재] 기술적 해자 주장과 모순**
   - 문제점: "기록 정확도 기술적 해자(Moat)"를 주장하면서 특허 출원, 기술 보호 전략이 전혀 없다. 앱별 파서 17개, Flow 감지 알고리즘, UIAutomation 활용 방식 등은 특허 또는 영업비밀로 보호 가능한 영역인데, GitHub에 오픈소스로 전체 공개되어 있어 해자가 오히려 무방비 상태.
   - 권장 수정: (1) 단기: GitHub 저장소를 private으로 전환하거나, 핵심 알고리즘(파서/Flow 감지) 부분만 분리. (2) 중기: 특허 출원 검토 (예비창업패키지 지재권 지원 활용 가능). (3) 사업계획서에 지재권 확보 계획 1~2줄 추가.
