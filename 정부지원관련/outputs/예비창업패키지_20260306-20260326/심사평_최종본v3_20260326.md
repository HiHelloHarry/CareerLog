### 실현가능성 상세 (16점)

수치 일관성에 치명적 문제가 있다.

18개월 목표로 MAU 10,000명, MRR 490만원을 제시했는데, 본인이 설정한 유료 전환율 7%를 적용하면 MAU 10,000 x 7% = 700명 유료, 700 x 4,900원 = MRR 343만원이다. MRR 490만원이 되려면 유료 1,000명이 필요하고, 이는 전환율 10%에 해당한다. 7%와 10% 사이의 갭을 어디서도 설명하지 않았다. 이런 산술 오류는 심사위원에게 "숫자를 검증하지 않았다"는 인상을 줄 수 있다.

해소 방안은 네 가지 중 하나를 선택해야 한다:
1. 전환율 목표를 10%로 상향하고, Premium(14,900원) 믹스를 반영한 ARPU 상승 논리 추가
2. MAU 목표를 14,286명으로 상향 (7% 유지)
3. MRR 목표를 343만원으로 하향 조정
4. Pro/Premium 구독자 믹스를 반영한 가중 ARPU(예: Pro 800명 x 4,900 + Premium 100명 x 14,900 = 540만원)로 설명

12개월 MRR 100만원은 MAU 3,000 x 7% = 210명 x 4,900 = 102.9만원으로 거의 맞지만, 18개월에서 갑자기 전환율이 뛰는 근거가 없다.

Phase 2 "Pro 플랜 출시, MAU 3,000명"이 6~12개월 구간이고, Phase 3 "MAU 10,000명"이 12~36개월 구간인데, 핵심 지표 테이블에서는 "12개월"과 "18개월"로 되어 있어 시점 매핑이 혼란스럽다.

### 성장전략 상세 (17점)

블라인드/링크드인 유기적 유입으로 MAU 3,000을 달성하겠다는 계획의 구체성이 부족하다.

- CAC(고객획득비용)를 전혀 제시하지 않았다
- 마케팅 예산 1,200만원을 6개월 쓰면 월 200만원인데, 이 예산으로 MAU 3,000을 만들려면 CAC가 4,000원 수준이어야 한다. SaaS에서 이건 매우 낙관적
- "블라인드 이직 카테고리에 실사용 후기 글 직접 작성"이 유일한 초기 그로스 전략인데, 이것으로 월 500명씩 신규 유저를 끌어오겠다는 건 비현실적
- B2B 확장을 "MAU 5,000 이상 달성 후"로 미뤘는데, B2B가 실질적 매출 드라이버가 될 텐데 시간축이 너무 김

### 팀구성 상세 (18점)

1인 창업자가 MVP까지 만든 건 인정한다. 하지만 지금 필요한 건 개발 능력이 아니라 유저를 데려올 능력이다. "그로스 공동창업자를 Phase 2 시작 전에 구한다"고 했는데, Phase 2 시작이 6개월 후다. 그때까지 그로스를 누가 하나?

공동창업자 후보 2명과 "탐색적 대화 진행 중"이라는 표현이 애매하다. 이름이나 배경을 밝힐 수 없더라도, "어떤 역량의 사람과 어떤 수준까지 논의됐는지" 정도는 있어야 한다.

### 총평

문제 정의와 제품 완성도는 예비창업 수준을 넘어선다. 설문으로 고객 수요를 검증하려는 시도도 좋다. 하지만 핵심 지표의 수치 불일치, 그로스 전략의 구체성 부족, 1인 팀 리스크가 아쉽다. "제품은 됐으니 시장 연결에 쓰겠다"는 포지셔닝은 좋은데, 그 시장 연결 계획이 가장 약하다.

**합격 의견:** 조건부 찬성 — 수치 일관성 수정 필수

### 실현가능성 상세 (21점)

기술 구현 수준이 예비창업 단계치고 상당히 높다. 구체적으로 확인하고 싶은 부분이 있다.

**1. Win32 API 직접 호출 방식의 기술적 판단**

active-win 네이티브 모듈의 asar 패키징 한계를 식별하고, PowerShell + Win32 API 방식으로 우회한 것은 엔지니어링 판단이 좋다. UIAutomationClient의 창 제목 오염 버그를 발견하고 GetForegroundWindow로 전환한 것도 실질적 기술 차별성이다. GitHub 커밋 이력으로 검증 가능하므로 신뢰할 수 있다.

**2. 파싱 성공률 100% 주장의 한계**

"가상 유저 시뮬레이션 3개 페르소나"에서 100%라고 했는데, 이건 테스트 케이스가 제한적이라 실환경 성공률과 다를 수 있다. 특히 한글 앱명이나 비표준 창 제목을 가진 기업용 소프트웨어(ERP, 사내 시스템 등)에서의 동작이 관건이다. 파서에 포함되지 않은 앱은 어떻게 처리되는지(fallback 로직) 설명이 없다.

**3. 10초 폴링 방식의 정보 손실**

10초 간격이면 짧은 앱 전환(5초 미만)은 누락된다. "53% 압축률"이 정보 손실 없는 압축인지, 정보 손실을 동반하는 압축인지 구분이 필요하다. 압축이라고 표현했지만 실제로는 정보 추상화(abstraction)에 가까울 수 있다.

**4. Mac 포팅 리스크**

Phase 2(6~12개월)에 Mac 개발 "시작"이라고 했는데, macOS에서 활동 감지는 Windows보다 권한 문제가 복잡하다. Accessibility 권한, 화면 녹화 권한 등이 필요하고, Apple의 보안 정책이 계속 강화되고 있다. osascript 기반이라고 했는데, osascript만으로는 브라우저 URL 추출이나 활동 감지가 Windows 수준으로 되지 않을 가능성이 높다.

**5. AI 프롬프트 품질의 제한적 검증**

STAR 변환 예시가 1건 제시되어 있고, Flow 기반 구조화 입력 + 역할 태그 설계가 체계적이다. 다만 "성과 수치 자동 추출"이라고 했는데 예시의 수치는 시간과 횟수뿐이다. ROI, 전환율 등 비즈니스 임팩트 수치를 자동 추출하는 건 PC 활동 감지만으로는 원천적으로 불가능하다. 이 한계를 명시하는 것이 오히려 신뢰를 높인다.

**6. 지재권 전략 부재**

기술 차별성을 강조하면서도 특허, 실용신안, 저작권 등록 등 지재권 확보 계획이 전혀 없다. Win32 API 기반 활동 감지 + Flow 감지 엔진의 조합은 충분히 특허 출원 대상이 될 수 있다. 예비창업 단계에서 출원까지는 어렵더라도 "어떤 기술 요소를 지재권화할 것인지"에 대한 인식이라도 보여줘야 한다.

### 성장전략 상세 (17점)

기술 로드맵 자체는 논리적이다. MVP(Windows) -> 결제 시스템 -> Mac -> JD 매칭 순서가 합리적이다. 하지만 Mac 포팅을 6~12개월 구간에 "시작"한다면서 외주 개발자(인도 또는 국내 프리랜서)에게 맡기는 건 리스크가 크다. macOS 활동 감지는 Windows와 완전히 다른 기술 스택이고, 이 도메인의 외주 개발 품질을 보장하기 어렵다.

### 팀구성 상세 (18점)

"기술 고문(비상근)"을 Phase 1에서 확보한다고 했는데, 어떤 분야의 기술 자문이 필요한지 구체적이지 않다. AI/ML인지, 인프라인지, 보안인지. 1인 개발 역량은 입증됐지만, 코드 리뷰를 받을 상대가 없다는 것은 기술 부채 누적 리스크다.

### 총평

기술 구현 수준이 예비창업 사업계획서 중에서 상위권에 해당한다. GitHub 소스 코드가 공개되어 있어 검증 가능하다는 점이 큰 강점이다. Win32 API 직접 호출, 앱별 파서 17개, Flow 감지 엔진 등은 단순한 래퍼가 아니라 실질적 기술 차별성이다. 다만 지재권 전략 부재, Mac 포팅 리스크 과소평가, 파싱 성공률의 제한적 검증 환경이 아쉽다.

**합격 의견:** 찬성

### 실현가능성 상세 (18점)

사업계획서의 가장 좋은 점은 "제품이 이미 있다"는 것이다. 예비창업 단계에서 MVP가 완성되어 있고, GitHub에서 소스 코드를 볼 수 있다는 건 진짜 드물다. 설문도 34명이 작지만, 질문이 구체적이고 결과 해석이 정직하다.

하지만 한 가지 걱정되는 게 있다. 설문에서 "바로 써보고 싶다"가 29%(10명)인데, 이 10명이 실제로 앱을 설치해서 일주일 이상 쓸 확률은 얼마일까? 설문의 "의향"과 실제 "행동" 사이의 갭을 이 사업계획서에서는 거의 다루지 않았다. 일반적으로 설문 의향 대비 실제 전환율은 10~30% 수준이다. 이걸 인정하면서도 "그래서 더 많은 모수가 필요하다"는 논리로 풀었으면 좋겠다.

D30 리텐션 40% 목표도 공격적이다. "매일 쓰는 습관 도구"라고 했는데, 솔직히 업무 시작/종료 버튼을 매일 누르는 것도 귀찮을 수 있다. 특히 재택이 아닌 사무실 출근 직장인은 PC를 켜는 것과 업무 시작이 일치하지 않는 경우가 많다. "귀찮음 제로"라고 했지만, 버튼 두 개도 귀찮은 사람은 분명히 있다. 이 이탈 시나리오에 대한 대응(예: PC 부팅 시 자동 시작, 캘린더 연동 자동 트리거)이 없다.

### 성장전략 상세 (16점)

초기 100명 모집 계획이 "블라인드에 글 쓰기 -> 커뮤니티 베타 초대 -> 지인 네트워크"인데, 구체적으로 물어보고 싶다:

- 블라인드에 글을 몇 건 쓸 건지? 어떤 톤으로, 어떤 카테고리에?
- 반응이 없으면 플랜B는?
- PM/기획자 오픈채팅방의 인원은 몇 명이고, 전환율 예상은?
- "비지인 채널 피드백을 의사결정 우선 기준으로 적용하여 편향 방지"는 좋은 문장인데, 비지인 채널에서 유저를 만나는 구체적 경로가 없다

잡코리아/원티드 파트너십도 "초기 접촉"이라고만 했는데, 이들이 왜 아직 시장에 없는 무명 서비스와 파트너십을 맺겠나? 이쪽은 B2C 트랙션이 먼저다.

### 팀구성 상세 (17점)

솔직하게 말씀드리면, 이 사업의 가장 큰 리스크는 1인 팀이다. 제품을 만든 건 대단하지만, 이제부터는 만드는 게 아니라 파는 단계다. 팀 확장 계획을 보면:

- 외주 개발자 (결제 시스템) — OK, 합리적
- 기술 고문 (비상근) — 좋은데, 실효성 의문
- 그로스 공동창업자 — "Phase 2 시작 전" (6개월 후)
- CS — "유료 300명 초과 시"
- 영업 — "MAU 5,000 달성 후"

전부 조건부다. "이 조건이 충족되면 그때 사람을 구하겠다"는 건데, 사람이 없어서 그 조건을 달성 못하면 어떡하나? 닭이 먼저냐 달걀이 먼저냐 문제다.

공동창업자 후보 2명이 있다면, 지금이라도 한 명을 "어드바이저" 형태로 참여시키는 게 심사에서 훨씬 유리하다. "탐색적 대화 진행 중"은 사실상 "아직 없다"와 같다.

### 총평

진정성과 제품 완성도는 높이 산다. 창업자가 자기 문제를 직접 풀었고, 설문으로 검증하려는 태도가 좋다. 하지만 "시장에 연결하는 단계"에서의 구체성이 부족하다. 베타 100명을 어떻게 모을 것인지, 그 100명이 왜 계속 쓸 것인지, 유료로 전환할 트리거가 무엇인지 — 이 세 질문에 대한 답이 지금 사업계획서에서는 약하다.

## 핵심 피드백 요약

### 강점

1. **제품 완성도가 예비창업 수준을 넘어선다.** MVP가 아니라 사실상 베타 직전 단계의 완성품이다. GitHub에서 소스 코드와 커밋 이력을 검증할 수 있어 기술 역량에 대한 신뢰도가 높다. Win32 API 직접 호출, 앱별 파서 17개, Flow 감지 엔진 등은 실질적 기술 차별성이다.

2. **설문 데이터의 질문 설계와 해석이 정직하다.** n=34는 통계적으로 작지만, 복수 질문을 통해 문제-솔루션-WTP를 체계적으로 검증했다. 외부 통계(경총 69.5%)와 자체 설문(76%)의 교차 검증, 자유 의견의 보안 우려를 솔루션과 연결한 논리가 설득력 있다.

3. **문제 정의의 프레이밍이 일관되고 강력하다.** "의지의 문제가 아니라 구조의 문제"라는 프레이밍이 사업계획서 전반에 일관 적용. "버튼 두 개로 매일 커리어가 쌓인다"는 한 줄이 솔루션의 본질을 정확히 전달한다.

### 보완 필수 항목 (치명적 약점)

1. **[수치 일관성] 18개월 MRR 목표와 유료 전환율 간 산술 불일치**
   - 문제점: MAU 10,000 x 전환율 7% = 유료 700명 x 4,900원 = MRR 343만원인데, 목표는 490만원(유료 1,000명, 전환율 10% 필요)
   - 권장 수정: (1) 전환율 7%를 유지하고 MRR을 343만원으로 하향, 또는 (2) Pro/Premium 믹스를 반영한 가중 ARPU로 설명, 또는 (3) 전환율 목표를 상향하되 근거 추가. 소요 시간 설문 비율 합계 101%도 100%로 조정 필요.

2. **[그로스 전략] 베타 100명 모집 및 MAU 성장 경로의 구체성 부족**
   - 문제점: "블라인드에 글 쓰기"를 넘어서는 채널 전략이 없음. CAC 미제시. 월간 신규 유입 목표 없음. 반응 없을 시 대안 없음.
   - 권장 수정: 주간 콘텐츠 발행 계획(채널별 1건/주 등), 채널별 기대 유입 수, CAC 추정(마케팅 1,200만원/목표 유저 수), 성과 미달 시 피벗 기준을 구체적으로 추가. 잡코리아/원티드 파트너십은 Phase 2 이후로 이동.

3. **[팀 리스크] 1인 창업자의 그로스 역량 공백 및 조건부 채용 계획**
   - 문제점: 모든 팀 확장이 "~달성 후" 조건부. 공동창업자 후보의 구체적 정보 없음. GTM 실행 주체 불명확.
   - 권장 수정: 공동창업자 후보의 역량/논의 수준을 기재하거나 어드바이저 LOI 확보. "Phase 1에서 창업자가 직접 수행할 그로스 활동" 주간 단위 계획 추가. 지재권 전략(특허 출원 검토 대상) 한 문단 추가.
