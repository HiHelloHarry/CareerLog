## 0. PD 검토 의견

### 이수진 초안 (UX)

**좋은 것:**
"결핍 지표 제외"와 "감정 흐름 설계"는 이 앱의 핵심 철학과 정확히 맞아떨어진다. Section A의 히어로 카드 개념, 빈 상태 3단계 분리, 문구 원칙은 그대로 가져간다. 탭 위치 제안("Timeline과 CareerResult 사이")도 정보 위계상 맞다.

**쳐낸 것:**
- 진입 경로 B(홈 카드 클릭)와 C(트레이 메뉴)는 v1에서 제외. 진입 경로를 분산시키면 사용자 학습 비용이 높아지고 구현 범위만 늘어난다. 탭 하나로 충분하다.
- Section E(앱 TOP 5)는 박민준 초안의 TOP 3와 중복. 3개로 정리.
- 진입 맥락별 행동 예측 테이블은 기획 사고 과정으로는 유효하나 개발 스펙에 포함할 내용이 아니다. 제거.

**보완:**
- 각 섹션의 데이터 소스와 업데이트 주기 명시.
- 히트맵 클릭 → Timeline 이동 시 날짜 파라미터 전달 방식 명시.

### 박민준 초안 (데이터)

**좋은 것:**
데이터 소스 3종 분리, 집계 로직, 스킬 정규화 원칙, 2단계 품질 판단 로직은 명확하고 바로 쓸 수 있는 수준이다.

**쳐낸 것:**
- "B2B/매각 시 활용 가능한 데이터 포인트" 섹션 전량 삭제. 전략 문서에서 다룰 것.
- "이력서 즉시 활용 수" 총량 카드 제거. "바로 쓸 수 있는 기록 12개 (전체 18개 중)"는 "6개는 못 쓴다"는 결핍 신호. CareerResult 배지로만 활용.
- 지표 감지 비율(metrics_detected %) 대시보드 노출 제외.
- "주 평균 활성 일수"는 "총 업무 일수"와 체감 차이 없음. 총 업무 일수 채택.
- 평균 세션 길이: 사용자 해석 불명확, v2에서 판단.

## 1. 기능 개요 및 목적

### 한 줄 정의
사용자가 지금까지 쌓아온 업무 기록을 한눈에 조망하는 커리어 현황 화면.

### 목적
CareerLog는 활동을 추적하고, Timeline은 오늘을 보여주고, CareerResult는 결과물을 보여준다. 대시보드는 이 모든 것을 연결해 "나는 얼마나 쌓았나"를 보여주는 유일한 화면이다. 사용자가 앱을 계속 쓸 이유를 만드는 것이 이 화면의 존재 이유다.

### 설계 원칙
1. **있는 것을 크게 보여준다.** 결핍 지표는 이 화면에 없다.
2. **숫자는 맥락과 함께.** "37일 동안 추적했어요"처럼 서술형으로.
3. **클릭은 항상 연결된다.** 모든 인터랙션 요소는 관련 화면으로 이동한다.

## 2. 화면 구성

### 탭 위치

현재: `Home / Timeline / CareerResult / Settings`
변경: `Home / Timeline / **대시보드** / CareerResult / Settings`

"내가 한 것(Timeline) → 전체 흐름(대시보드) → 결과물(CareerResult)" 정보 위계.

---

### Section A — 이번 주 활동 요약 (히어로 카드)

**위치:** 최상단, 스크롤 없이 보이는 첫 번째 블록

```
┌─────────────────────────────────────┐
│  이번 주에 13.1시간 일했어요           │  ← 24px+, 메인 타이포
│                                     │
│  월 ████░░░  4.2h                   │
│  화 ██████░  5.8h                   │
│  수 ███░░░░  3.1h                   │
│  목 ─────── (오늘, 아직 기록 없음)    │
│                                     │
│  가장 많이 한 일: Figma (3.2h)       │
└─────────────────────────────────────┘
```

**문구 규칙:**

| 상황 | 표시 문구 |
|---|---|
| 이번 주 기록 있음 | "이번 주에 [N]시간 일했어요" |
| 월요일, 기록 0 | "이번 주를 시작했어요" |
| 이번 주 기록 없음 (화~일) | 지난주 데이터 폴백 + "지난주 기록이에요" 레이블 |
| 지난주도 없음 | "이번 주 업무를 시작해보세요" |

**데이터 소스:** sessions.json (이번 주 start_time 기준), activities.json (앱별 시간)

---

### Section B — 누적 커리어 현황 (2×2 카드)

```
┌───────────────┐  ┌───────────────┐
│  총 업무 일수  │  │  경력 기록 수  │
│  37일 동안     │  │  12개 생성     │
└───────────────┘  └───────────────┘
┌───────────────┐  ┌───────────────┐
│  총 추적 시간  │  │  다룬 앱/도구  │
│  184시간       │  │  23가지        │
└───────────────┘  └───────────────┘
```

**지표 정의:**

| 레이블 | 집계 기준 | 데이터 소스 |
|---|---|---|
| 총 업무 일수 | 활동 기록 1건 이상인 날의 고유 일수 | activities.json → timestamp 날짜 중복 제거 |
| 경력 기록 수 | career_records 전체 개수 | career_records.json |
| 총 추적 시간 | 완료 세션의 end_time - start_time 합산 | sessions.json (완료 세션만) |
| 다룬 앱/도구 | 고유 앱 이름 수 (CareerLog·시스템 프로세스 제외) | activities.json → appName 중복 제거 |

**기간 필터:** 비적용. 항상 전체 누적 기준. (숫자가 줄어드는 경험 방지)

---

### Section C — 활동 히트맵

GitHub 잔디 스타일. 최근 12주(84일) 기준.

**스펙:**
- 셀 단위: 1일
- 색상 농도 4단계: 0시간(회색) / 1시간 미만 / 1~3시간 / 3시간 초과
- 호버: `[날짜] · [N]시간 추적` 툴팁 (100ms 딜레이)
- 클릭: 해당 날짜 파라미터를 Timeline에 전달하여 이동 (활동 없는 날 비활성)
- 범례: 하단 `적음 ◻◻◻◻ 많음`
- 오늘: 셀 테두리 강조

**데이터 소스:** sessions.json (start_time 기준 날짜 집계)

---

### Section D — 최근 경력 기록 미리보기

가장 최근 생성된 경력기록 2개.

**카드 1개당:**
- 생성 날짜
- bullets[0] 1줄 (말줄임)
- `자세히 보기` → CareerResult 해당 기록 포커스 이동

**하단:** `전체 경력 기록 보기 →` → CareerResult 이동

**데이터 소스:** career_records.json (created_at 내림차순 상위 2건)

---

### Section F — 스킬 태그 클라우드

**스펙:**
- 최대 20개 (빈도 상위, 최소 출현 2회 이상)
- 폰트 크기 3단계: 상위 10%(대) / 11~40%(중) / 나머지(소)
- 정규화: 소문자 변환, 공백 trim, 중복 제거
- 클릭: 해당 스킬 포함 경력기록 필터 → CareerResult 이동
- 하단 요약: `총 [N]개 스킬 · 최다 등장: [스킬명]`

**데이터 소스:** career_records.json → skills[] 플랫 후 집계

**기간 필터:** 적용 (career_records.created_at 기준)

---

### Section G — 많이 쓴 앱 TOP 3

**스펙:**
- 가로 바 차트: 앱명 + 총 사용 시간
- 드롭다운: 이번 주 / 이번 달(기본) / 전체
- 클릭: Timeline 해당 앱 필터 자동 적용 이동
- CareerLog 자신 및 시스템 프로세스 제외

**데이터 소스:** activities.json → appName별 duration 합산

## 3. 기간 필터 적용 범위

| 섹션 | 기간 필터 | 비고 |
|---|---|---|
| Section A (이번 주 요약) | 비적용 (이번 주 고정) | |
| Section B (누적 카드) | 비적용 (전체 누적 고정) | 숫자 감소 경험 방지 |
| Section C (히트맵) | 비적용 (최근 12주 고정) | |
| Section D (경력기록 미리보기) | 비적용 (최신 2건 고정) | |
| Section F (스킬 클라우드) | 적용 (career_records.created_at) | |
| Section G (앱 TOP 3) | 적용 (섹션 내 독립 드롭다운) | |

**전역 기간 필터 v1 미구현.** 섹션마다 의미가 달라 하나의 필터로 묶으면 숫자가 줄어드는 섹션이 반드시 생긴다.

## 4. 빈 상태 처리

| 상태 | 조건 | 처리 방식 |
|---|---|---|
| 완전 초기 | 활동 0건, 세션 0건 | 전체 화면을 온보딩 유도로 대체. 문구: "업무를 시작하면 여기에 쌓이기 시작해요" + "지금 업무 시작하기" CTA |
| 부분 데이터 | 활동 있음, career_records 0건 | Section D·F만 빈 상태. 나머지 정상 렌더링 |
| 이번 주 활동 없음 | 이번 주 sessions 0건 | Section A: 지난주 데이터 폴백 + "지난주 기록이에요" 레이블 |
| 스킬 태그 부족 | 출현 2회 이상 스킬 없음 | Section F: "경력 기록이 쌓이면 자주 쓴 스킬이 보여요" |

**빈 상태 문구 원칙:** "아직 기록이 없어요" 금지. 항상 미래 행동 유도형으로.

## 5. 구현 우선순위

### P0 — 필수

| 항목 | 근거 |
|---|---|
| 탭 추가 및 화면 라우팅 | 진입점 없으면 아무것도 안 됨 |

### P1 — 동일 또는 직후 스프린트

| 항목 | 근거 |
|---|---|
| Section C 히트맵 | "잔디 심기" 감각이 재방문을 유도하는 핵심 |
| Section D 경력기록 미리보기 | CareerResult 유입 유도 |
| Section G 앱 TOP 3 | 데이터 있으면 바로 쓸 수 있음 |
| 부분 빈 상태 처리 | P0 완성 후 즉시 필요 |

### P2 — 다음 스프린트

| 항목 | 근거 |
|---|---|
| Section F 스킬 태그 클라우드 | 경력기록 충분히 쌓여야 의미 있음, 초기엔 빈 섹션 가능성 높음 |
| 히트맵 → Timeline 날짜 연동 | 파라미터 전달 구현 별도 필요 |
