## 1. PD 리뷰 노트

> 이 섹션은 v0.1 기획서 → v0.2 기획서 전환 과정에서 PD가 발견한 핵심 문제점과 수정 방향을 요약한다. 개발팀과 기획팀이 "왜 이렇게 바뀌었는가"를 추적할 수 있도록 보존한다.

### 1-1. 치명적 문제 (즉시 수정)

**[문제 1] 수익 모델 부재**

v0.1 기획서에 수익화에 대한 언급이 단 한 줄도 없다. "Claude API 키를 사용자가 직접 입력"하는 구조는 개발자/얼리어답터에게만 작동하는 모델이다. 일반 직장인이 Anthropic Console에 가입하고 API 키를 발급받을 전환율은 현실적으로 5% 미만이다. 제품의 핵심 기능이 API 키 없이는 불가능한 구조에서, 이 진입 장벽을 어떻게 처리할지가 제품 생사를 결정한다.

**수정 방향:** 섹션 4에 비즈니스 모델 전략 신설. Freemium 구조 설계 (기본 추적/로컬 저장은 무료, AI 생성은 크레딧 기반 유료).

**[문제 2] 온보딩 플로우 전무**

처음 설치한 사용자가 "업무 시작" 버튼을 누르기까지 무엇을 경험하는지 설계가 없다. API 키 없이 앱을 켰을 때 사용자는 앰버 배너 하나를 본다. 이 사용자가 API 키를 발급받으러 Anthropic 콘솔로 가는 여정을 아무도 설계하지 않았다. Day 1 이탈의 주요 원인이 될 것이다.

**수정 방향:** 섹션 5에 온보딩 플로우 3단계 상세 기획 신설.

**[문제 3] 데이터 유실 위험**

로컬 JSON 전용 저장 구조에서 PC 포맷, 기기 교체, 파일 손상 시 수개월치 경력 기록이 한 번에 소멸된다. 이 리스크를 기획서가 인지조차 하지 않고 있다. 사용자가 2~3개월 기록한 후 데이터를 잃으면 앱에 대한 신뢰가 영구적으로 손상된다.

**수정 방향:** 섹션 6에 데이터 관리 전략 신설. 단기는 export/import, 중기는 선택적 클라우드 백업.

### 1-2. 중요 문제 (v0.2에서 수정)

**[문제 4] Streak을 P0로 분류한 것은 과잉**

P0 기준은 "없으면 제품이 안 되는 것"이다. Streak은 리텐션 강화 도구이지, 제품의 핵심 가치가 아니다. STAR 구조와 Done 화면은 P0가 맞다. Streak은 P1로 재분류한다.

**수정 방향:** Gap 분석 우선순위 재조정.

**[문제 5] 경쟁 분석이 2019년 수준**

Notion/LinkedIn만 경쟁사로 언급했다. 2025-2026년 AI 이력서 시장에는 Teal HQ, Rezi, Kickresume AI, Resume.io, 그리고 자동 시간 추적 관점에서 RescueTime, Toggl Track이 있다. 이들과의 차별화 포인트가 명확히 설계되지 않으면 포지셔닝이 흐려진다.

**수정 방향:** 제품 개요 경쟁 분석 섹션 전면 재작성.

**[문제 6] active-win 기술 한계 미인지**

active-win은 앱 이름과 창 제목(window title)만 감지한다. Chrome에서 어떤 탭에 있었는지 알 수 없다. "Chrome — 58분"이 경력 기록에 어떻게 기여하는가? 브라우저 탭 구분 불가 한계를 기획서가 명시하고, UX에서 메모로 보완하도록 유도해야 한다. 또한 30초 폴링은 빠른 컨텍스트 스위칭(10~15초)을 놓친다.

**수정 방향:** 기술 스택 섹션에 한계 명시, 메모 기능을 더 강조.

**[문제 7] Windows/macOS 동작 차이 무시**

active-win은 macOS에서 accessibility 권한 필요, Windows에서는 UAC 이슈가 있다. 기획서에 플랫폼별 차이 언급이 없다. 초기 버전이라도 어느 플랫폼을 우선 지원하는지 명시해야 한다.

**수정 방향:** 기술 스택에 플랫폼 우선순위 명시.

## 2. 제품 개요

### 2-1. 한 줄 정의

> **일하는 순간마다 경력이 자동으로 쌓이는 데스크탑 앱 — 이직할 때 꺼내 쓰면 된다.**

### 2-2. 제품이 해결하는 문제

대부분의 직장인은 퇴직 직전에야 이력서를 쓰려 한다. 그 순간 3~5년간 자신이 무슨 일을 했는지 구체적으로 기억하지 못한다. 노션·옵시디언 같은 도구로 기록을 시도하더라도 구조 없이 흩어지거나, 귀찮아서 결국 멈춘다.

핵심 문제는 세 가지다:

1. **기억의 한계**: 3개월 전 어떤 프로젝트에서 무슨 기여를 했는지 기억하지 못한다
2. **기록의 마찰**: 노션에 경력을 기록하는 행위 자체가 업무 외 추가 노동이다
3. **구조화의 어려움**: 기억이 있더라도 이력서용 STAR 구조로 변환하는 능력이 부족하다

CareerLog는 이 세 문제를 모두 해결한다. 추적은 자동으로, 구조화는 AI가, 사용자는 검토와 수정만 한다.

### 2-3. 경쟁 분석 (2026년 기준)

#### 직접 경쟁

| 제품 | 카테고리 | 강점 | 한계 | CareerLog 차별점 |
|---|---|---|---|---|
| Teal HQ | AI 이력서 관리 | 구직 파이프라인 통합 | 수동 입력 필요 | 자동 추적으로 기억 의존 제거 |
| Rezi | AI 이력서 작성 | ATS 최적화 | 있는 경력 정리만, 기록 기능 없음 | 경력 기록 자체를 생성 |
| Kickresume AI | 이력서 생성 | 디자인 품질 높음 | 기록 기능 없음 | 일상 업무에서 직접 추출 |
| Resume.io | 이력서 편집기 | 간편한 UI | AI 생성 품질 보통 | STAR 구조 자동화 |

#### 간접 경쟁 (시간 추적 도구)

| 제품 | 카테고리 | 강점 | 한계 | CareerLog 차별점 |
|---|---|---|---|---|
| RescueTime | 생산성 추적 | 자동 앱 추적 | 경력 기록 기능 없음, 목적이 다름 | 추적 → 경력 기록 변환이 핵심 |
| Toggl Track | 시간 추적 | 프로젝트별 관리 | 수동 시작/종료 필요 | 완전 자동 추적 |
| Wakatime | 개발자 코딩 시간 | IDE 플러그인 | 개발자만 사용 가능 | 모든 직군 지원 |

#### 핵심 차별화 포인트

```
CareerLog = 자동 추적(RescueTime) + AI 경력 변환(Rezi) + 기록 누적(Notion)
```

경쟁사들은 각각 하나의 문제만 해결한다. CareerLog는 "기록 → 구조화 → 축적"의 전체 파이프라인을 하나의 앱에서 해결한다.

### 2-4. 타겟 유저 정의

**Primary Persona: "기억력에 지친 5년차 PM 김민준"**

- 나이: 28~35세
- 직군: PM, 기획자, 디자이너, 마케터 (비개발 직군 우선)
- 상황: 이직을 고민하기 시작했지만 이력서 쓰기가 막막함
- 페인포인트:
  - "내가 뭘 했는지는 아는데, 어떻게 쓸지 모르겠다"
  - "수치를 기록해 두지 않아서 이력서에 쓸 게 없다"
  - "노션에 기록하려다 3일 만에 포기했다"
- 기술 수준: 앱 설치는 가능하나 API 키 발급은 어렵게 느낌 → **온보딩이 핵심**

**Secondary Persona: "사이드 프로젝트 병행 개발자 박지수"**

- 나이: 25~32세
- 직군: 프론트엔드/백엔드 개발자
- 상황: 본업 + 사이드 프로젝트를 동시에 진행, 다음 이직 시 포트폴리오 정리 필요
- 페인포인트: 어떤 기술을 얼마나 썼는지 정리가 안 됨
- 기술 수준: API 키 발급 가능 → **기능 깊이가 중요**

**Excluded Persona (현재 단계에서 제외)**

- 취준생 신입: 경력이 없어 추적할 내용이 없음
- C레벨 임원: 업무 기록 방식이 다르고, 이력서 작성을 직접 안 함
- 영업직: 앱 사용 패턴이 다르고 추적 가능한 디지털 활동이 적음

### 2-5. 핵심 가치 제안 (Value Proposition)

```
[For] 이직을 준비하는 직장인 3~7년차
[Who] 매일의 업무 성과를 체계적으로 기록하지 못해 이력서 작성 시 막막함을 느끼는
[CareerLog is] 데스크탑 자동 추적 + AI 경력 변환 앱
[That] 오늘 일한 것을 자동으로 감지하여 이력서용 STAR 구조 경력 기록으로 변환한다
[Unlike] Notion(수동), Rezi(경력 있어야 사용), RescueTime(경력 변환 불가)
[Our product] 기록의 마찰을 없애고 AI가 구조화를 대신해 준다
```

### 2-6. 솔루션 플로우

```
[시스템 트레이 상주]
       ↓
업무 시작 → 30초마다 활성 앱/창 자동 감지
       ↓
업무 종료 → 타임라인 리뷰 + 메모 추가
       ↓
Claude AI → STAR 구조 경력 기록 생성
       ↓
경력 저장소 누적 → 이력서/LinkedIn/자기소개서 출력
```

## 3. 현재 버전 기능 정의 (v0.1)

v0.1에서 구현 완료된 기능을 개발자가 유지보수하거나 v0.2를 이어서 작업할 때 기준이 될 수 있도록 상세히 기술한다.

### 3-1. 시스템 트레이 앱

**동작 방식**
- Electron 앱이 시작되면 시스템 트레이에 아이콘이 등록된다.
- 트레이 우클릭 메뉴: `▶ 업무 시작` / `⏹ 업무 종료` / `창 열기` / `종료`
- 트레이 더블클릭 시 메인 창이 열린다.
- 기록 중일 때 트레이 툴팁이 `CareerLog — 기록 중`으로 변경되고, 가장 최근 활성 앱명이 메뉴에 표시된다.
- `app.on('window-all-closed')` 이벤트를 막아 창을 닫아도 앱이 종료되지 않도록 처리됨.

**파일 위치:** `electron-app/src/main.js`

### 3-2. 홈 화면 (Home)

**비기록 상태 (isTracking = false)**
- API 키 미설정 시 상단에 앰버색 배너 표시 → 클릭 시 설정 탭으로 이동
- 프로젝트/클라이언트명 입력 필드 (선택, Enter 키로 시작 가능)
- `업무 시작` 버튼 — 클릭 시 `window.careerlog.startTracking(project)` IPC 호출
- 확인 오버레이 출력: "업무 기록이 시작되었습니다" + 트레이 사용법 안내
- `확인, 창 닫기` 버튼 → `window.close()`

**기록 중 상태 (isTracking = true)**
- 경과 시간 타이머 (HH:MM:SS, 초 단위 카운트업)
- 파란색 맥동 애니메이션 "기록 중" 표시
- `업무 종료` 버튼 → `window.careerlog.stopTracking()` → 타임라인 화면으로 전환

**파일 위치:** `electron-app/src/components/Home.jsx`

### 3-3. 활동 추적 (Tracker)

**동작 방식**
- `startTracking()` 호출 시 `setInterval`로 30초마다 `active-win` 실행
- 반환값: `{ owner: { name }, title }` → app_name, window_title로 저장
- 마지막 활동과 앱명이 같으면 체류 시간 연장, 다르면 새 활동 레코드 생성
- `stopTracking()` 호출 시 마지막 활동의 ended_at을 현재 시각으로 flush

**저장 구조 (로컬 JSON)**
```
sessions.json     — 세션 목록 (id, date, started_at, ended_at, project)
activities.json   — 활동 목록 (id, session_id, app_name, window_title,
                               started_at, ended_at, duration_sec, memo)
```

**파일 위치:** `electron-app/src/tracker.js`, `electron-app/src/database.js`

### 3-4. 타임라인 화면 (Timeline)

**ActivityItem 구성 요소**

- 앱명 + 체류 시간 칩
- 창 제목 (truncate 처리)
- 시작/종료 시각 (HH:MM 포맷)
- 병합 표시: `🔗 N개 통합` 칩 (merged_ids.length > 1인 경우)
- 메모 입력: 없으면 `+ 메모 추가`, 있으면 앰버색 칩으로 표시 → 클릭 시 인라인 편집

**활동 병합 로직**

- 병합된 경우 앰버색 테두리 + `🔗 N개 통합` 칩으로 시각적 구분

**파일 위치:** `electron-app/src/components/Timeline.jsx`

## 4. 비즈니스 모델 & 수익화 전략

> 이 섹션은 v0.1에 완전히 누락된 신규 섹션이다. 수익 경로 없이 제품을 지속 운영하는 것은 불가능하며, 특히 Claude API 비용이 발생하는 구조에서 비즈니스 모델 설계는 필수다.

### 4-1. 현재 구조의 문제점

**"API 키 직접 입력" 모델의 현실**

현재 CareerLog는 사용자가 Anthropic API 키를 직접 발급받아 입력해야 AI 기능을 사용할 수 있다. 이 모델의 전환율 문제를 직시해야 한다:

| 단계 | 예상 이탈률 | 이유 |
|---|---|---|
| 앱 다운로드 | - | 기준점 |
| 앱 설치 완료 | -15% | 설치 과정 마찰 |
| API 키 발급 시도 | -40% | Anthropic Console 가입 필요 |
| API 키 발급 완료 | -25% | 결제 수단 등록, 영문 인터페이스 |
| API 키 입력 완료 | -10% | 복사/붙여넣기 과정 실수 |
| **첫 AI 경력 기록 생성** | **약 10~15%만 도달** | |

API 키 직접 입력 모델을 유지하면 실제 핵심 가치를 경험하는 사용자가 전체의 10~15%에 그친다. 이것은 제품 가치가 아닌 진입 장벽 때문에 사용자를 잃는 것이다.

### 4-2. 권장 비즈니스 모델: Freemium + AI 크레딧

#### 핵심 전략

```
무료 티어: 추적 + 로컬 저장 + 제한적 AI 생성
유료 티어: 무제한 AI 생성 + 클라우드 백업 + 고급 출력
```

**무료 티어 (Free)**

| 기능 | 무료 제공 여부 |
|---|---|
| 자동 앱 추적 | ✅ 무제한 |
| 로컬 JSON 저장 | ✅ 무제한 |
| 타임라인 뷰 | ✅ 무제한 |
| 수동 메모 입력 | ✅ 무제한 |
| AI 경력 기록 생성 | ✅ 월 5회 (자체 크레딧) |
| STAR 구조 생성 | ✅ 월 5회 포함 |
| 데이터 export (JSON) | ✅ 무제한 |
| 클라우드 백업 | ❌ |
| 주간 AI 요약 | ❌ |
| 영문 이력서 출력 | ❌ |
| PDF 직접 생성 | ❌ |

**유료 티어: CareerLog Pro (월 9,900원 / 연 89,000원)**

| 기능 | Pro |
|---|---|

| 우선 고객 지원 | ✅ |

**API 키 직접 입력 옵션 유지**

얼리어답터/개발자를 위해 "내 API 키 사용" 옵션은 유지한다. 단, 기본 경험은 자체 크레딧 기반으로 제공하여 진입 장벽을 제거한다.

```
설정 화면:
  [ ] 자체 크레딧 사용 (기본, 월 5회 무료)
  [ ] 내 API 키 사용 (무제한, 비용 직접 부담)
  → Pro 구독 시 자체 크레딧 무제한
```

### 4-3. 수익 모델 단계별 전환 경로

**Phase 1 (v0.2~v0.3): 제품-시장 적합성 탐색**

- 유료 전환 유도 트리거: "이번 달 무료 AI 생성 5회를 모두 사용했습니다"
- 목표 전환율: 5% → 50명 유료 사용자

**Phase 2 (v1.0): 수익화 본격화**

- 유료 사용자 500명 → 월 495만원 MRR 목표
- 연간 구독 할인으로 현금 흐름 안정화
- 기업 플랜 탐색 (팀 단위 사용 — 인사팀이 구성원 경력 관리)

**Phase 3 (v1.x): 플랫폼화**

- 채용 플랫폼 연동 수익 (LinkedIn, 원티드 API)
- 경력 데이터 기반 커리어 코칭 서비스 연계
- B2B: 기업 HRD 팀 라이선스

## 6. 데이터 관리 전략

> 이 섹션은 v0.1에 완전히 누락된 신규 섹션이다. 로컬 JSON 저장 구조의 위험성을 인지하고, 단계별 데이터 보호 전략을 수립한다.

### 6-1. 현재 데이터 저장 구조의 위험성

**저장 위치:** Electron app의 `userData` 경로
- Windows: `C:\Users\{username}\AppData\Roaming\CareerLog\`
- macOS: `~/Library/Application Support/CareerLog/`

**데이터 유실 시나리오:**

| 시나리오 | 발생 가능성 | 영향 |
|---|---|---|
| PC 포맷 / 초기화 | 중간 | 전체 경력 기록 소멸 |
| 기기 교체 | 높음 | 새 PC에서 기록 없음 |
| 앱 삭제 후 재설치 | 중간 | userData 폴더 포함 삭제 시 소멸 |
| 파일 손상 (JSON 파싱 오류) | 낮음 | 특정 날짜 기록 접근 불가 |

3개월 이상 기록한 사용자가 데이터를 잃으면 제품에 대한 신뢰가 영구 손상된다. 이것은 기능이 아닌 신뢰의 문제다.

  star JSONB,
  created_at TIMESTAMPTZ,
  -- Row Level Security: 자신의 데이터만 접근
  CONSTRAINT rls_policy CHECK (user_id = auth.uid())
);

## 8. v0.2 상세 기획

### 8-1. 구조화된 STAR 경력 기록 — P0

#### 기능 설명

현재 AI 출력은 자유 텍스트 bullet 형태다. v0.2에서는 Claude가 다음 5개 필드를 가진 구조화된 JSON을 반환하도록 프롬프트를 개선하고, UI에서 각 필드를 별도 카드로 렌더링한다.

| 필드 | 설명 | 예시 |
|---|---|---|
| **Situation** | 배경/상황 | "신규 사용자 온보딩 전환율이 31%로 업계 평균 대비 낮은 상황" |
| **Task** | 맡은 역할/목표 | "온보딩 플로우 전면 재설계 리드 (PM 1인, 디자이너 1인)" |
| **Action** | 구체적 행동 | "사용자 인터뷰 6건, 경쟁사 벤치마킹 → 3단계 → 2단계로 단순화" |
| **Result** | 결과/성과 | "온보딩 전환율 31% → 58% 달성 (3개월 내)" |
| **Skills** | 사용 기술/스킬 | `["UX Research", "Figma", "A/B Testing", "Jira"]` |

#### 경력 템플릿 (확장)

STAR 외에 목적별 경력 기록 템플릿을 제공한다. 사용자가 생성 전 템플릿을 선택할 수 있다.

| 템플릿 | 특징 | 적합 상황 |
|---|---|---|
| **STAR (기본)** | Situation-Task-Action-Result 구조 | 일반 이직 이력서 |
| **숫자 중심** | 수치 성과를 전면에 배치, 달성률/개선율 강조 | 성과 중심 이력서 (마케터, 사업개발) |
| **프로세스 중심** | 어떤 방법론/프로세스로 문제를 해결했는지 | 기획자, PM, 컨설턴트 |
| **결과 중심** | 비즈니스 임팩트와 아웃컴을 중심으로 기술 | 시니어급, C레벨 지원 |

```
타임라인 화면 → "경력 기록 생성" 버튼 클릭 시:

  어떤 형식으로 기록하시겠어요?

  ○ STAR 구조 (기본)
  ○ 숫자/성과 중심
  ○ 프로세스 중심
  ○ 결과 중심

  [생성하기]
```

1. 타임라인에서 `경력 기록 생성` 클릭
2. 템플릿 선택 모달 표시 (기본: STAR)

7. 각 필드 옆 `편집` 아이콘 클릭 → 인라인 편집
8. Skills 태그 리스트 표시 + 태그 추가/삭제 가능
9. 수치 발견 시 Result 카드에 `수치 발견!` 골드 뱃지

### 8-2. 수치 자동 감지 & 하이라이트 — P0

#### 기능 설명

창 제목, 메모, AI 생성 텍스트에서 숫자가 포함된 성과 지표를 자동으로 감지하여 강조 표시한다.

**감지 대상 패턴**

| 패턴 유형 | 예시 |
|---|---|
| 퍼센트 | `전환율 31% → 58%`, `오류율 75% 감소` |
| 절대 수치 + 단위 | `MAU 12만`, `DAU 3,200`, `사용자 1,000명` |
| 금액 | `매출 2억`, `비용 30% 절감` |
| 시간 단위 성과 | `3주 만에`, `배포 주기 2주 → 3일` |
| 배율 | `2배 성장`, `3x 개선` |

#### 구현 방식

```javascript
const METRIC_PATTERNS = [
  /\d+[%％]/g,
  /\d+[만억천백]+/g,
  /\d{1,3}(,\d{3})+/g,
  /\d+[xX배](\s|$)/g,
  /\d+\s*(명|건|회|개|일|주|달|년)/g,
];

// 날짜/시각 패턴 제외
const EXCLUDE_PATTERNS = [
  /\d{4}-\d{2}-\d{2}/,
  /\d{2}:\d{2}/,
  /v\d+\.\d+/,
];

function detectMetrics(text) {
  const found = [];
  METRIC_PATTERNS.forEach(pattern => {
    const matches = text.match(pattern);
    if (matches) found.push(...matches);
  });
  // 제외 패턴 필터
  return [...new Set(found)].filter(m =>
    !EXCLUDE_PATTERNS.some(p => p.test(m))
  );
}
```

#### 엣지 케이스

| 상황 | 처리 방식 |
|---|---|
| 수치 없는 세션 | 수치 관련 UI 요소 미표시 |
| 날짜/시각이 수치로 감지 | 제외 패턴으로 필터링 |
| 파일 버전 (`v2`, `_3`) | 단독 숫자 아닌 경우 제외 |

## 9. 성공 지표 & 측정 방법

> 이 섹션은 v0.1에 완전히 누락된 신규 섹션이다. KPI 없이 제품을 개선하는 것은 나침반 없이 항해하는 것과 같다.

### 9-1. 핵심 지표 (North Star)

**North Star Metric: 월간 경력 기록 생성 수 (Monthly Career Records Created)**

왜 이것인가:
- CareerLog의 핵심 가치는 "경력 기록 생성"이다
- 이 수치가 높다는 것은 사용자가 실제로 제품을 사용하고 있음을 의미한다
- DAU보다 더 의미 있는 지표 (앱을 켰지만 기록 안 한 것과 구분)

**목표 (v0.2 출시 후 3개월):**
- 월간 경력 기록 생성 수: 사용자당 평균 8개/월 (주 2개)
- 총 경력 기록 생성 수: 2,000개/월 (사용자 250명 기준)

### 9-2. 제품 건강 지표

| 지표 | 설명 | 목표 | 측정 방법 |

| **D1 리텐션** | 설치 다음 날 재실행률 | 40% | 앱 실행 이벤트 |

### 부록 D. 주요 화면 플로우 (v0.2)

```
[첫 실행]
  Onboarding Welcome
    → AI 설정 (무료 크레딧 or API 키)
      → 첫 기록 시작 (업무 시작 or 데모)
        → [업무 추적 중]
          → 업무 종료
            → Timeline (타임라인 리뷰)
              → 템플릿 선택 → AI 생성 중
                → Done 화면
                  → CareerResult (STAR 카드)

[일반 실행]
  Home (스트릭 바 표시)
    → [업무 시작]
      → [백그라운드 추적]
        → [업무 종료]
          → Timeline
            → [경력 기록 생성]
              → Done → CareerResult

[수동 입력]
  Home → [직접 입력하기]
    → ChatInput (AI 대화)
      → Done → CareerResult

[히스토리 탐색]
  NavRail → Calendar
    → 날짜 선택
      → CareerResult (해당 날짜 기록)
```
